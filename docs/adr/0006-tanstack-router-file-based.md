# フロントエンドルーティング: TanStack Router（ファイルベースルーティング）

フロントエンドのルーティングに TanStack Router のファイルベースルーティングを採用する。ルート定義がファイルシステムの構造で決まるため、Lint やスクリプトで構造を機械的に検証でき、型安全なパスパラメータ・検索パラメータが得られる。

## Considered Options

| 選択肢 | 不採用理由 |
|---|---|
| React Router | ファイルベースルーティングがオプション扱い。パスパラメータ・検索パラメータの型推論が TanStack Router より弱い。ルート定義が分散しやすく Lint で構造を強制しにくい |
| Next.js App Router | フルフレームワーク。バックエンドが Spring Boot なので SSR/RSC は不要。Vite ベースのプロジェクトに導入できない |
| Wouter | 軽量だが型安全性なし。ファイルベースルーティングなし。業務アプリの規模には不向き |
| TanStack Router（コードベースルーティング） | 手動でルート定義を書く方式。ファイルベースの方がディレクトリ構造＝ルート構造となり、規約の機械的検証が容易 |

## 採用理由

- **ファイル構造 = ルート構造**: ディレクトリを見ればルーティングが分かる。新規ルート追加時に設定ファイルの編集が不要
- **型安全**: パスパラメータ・検索パラメータが TypeScript の型として推論される。存在しないルートへの遷移がコンパイルエラーになる
- **Lint 統制との相性**: ファイル命名規則・ディレクトリ構造をスクリプトや Lint で検証可能。「ルートファイルに副作用のあるロジックを書かない」等のルールを強制できる
- **データローダー内蔵**: ルートごとにデータ取得ロジックを宣言的に定義でき、コンポーネントとデータ取得の責務が分離される

## Consequences

- `frontend/src/routes/` 配下のファイル構造がそのままルーティングになる
- TanStack Router の CLI（`@tanstack/router-plugin`）が Vite プラグインとしてルート定義を自動生成する
- ルートファイルの命名規則（`$param` でパスパラメータ、`_layout` でレイアウト等）はフレームワークの規約に従う
